# CLAUDE.md — model-harness

## What this is

**SapphireGuard.ModelHarness** is a .NET 10 agent framework that wraps LLMs with structured scaffolding. An *agent = model + harness* — the harness (loop, guides, sensors, budget) is what turns a raw model into a controllable agent.

## Build & run

```bash
dotnet build SapphireGuard.ModelHarness.slnx
dotnet run --project samples/HappyPath
```

Add an API key first (per-sample; samples fall back to `FakeModelClient` when absent):

```json
// samples/HappyPath/appsettings.local.json
{ "Anthropic": { "ApiKey": "sk-ant-..." } }
```

## Core patterns

### HarnessLoop
Turn-by-turn state machine: check budget → sensors (PreModelCall) → build context via guides → call model → sensors (PostModelCall) → dispatch tools → sensors (Pre/PostToolCall) → repeat. Budget exhaustion is control flow, not an exception — one final model call with tools disabled, then `PartialResult`. An optional per-tool-call deadline (`Budget.MaxToolCallDuration`) wraps each dispatch in a `TimeProvider`-based linked `CancelAfter`, turning a timeout into a recoverable `IsError` result rather than a run failure (cooperative — bounds tools that honour their token). Model calls have no second harness deadline: the provider SDKs apply their own request timeout, tunable via the adapter `ConfigureClient` hook.

### Guide pattern
Guides run **sequentially** before each model call, each contributing to a shared `ContextDraft`. Pipeline order is explicit and fixed: `HarnessInstructionsGuide` → `ReActGuide` → `MemoryGuide` → `ToolSelectorGuide` → `ToolCatalogueGuide` → `SkillsGuide` → `PinnedContextGuide` → [custom guides] → `HeadEvictionTrajectoryGuide`. `HeadEvictionTrajectoryGuide` is always last so it can measure all prior contributions and compute an accurate token budget. `ToolSelectorGuide` must precede `ToolCatalogueGuide` — the catalogue renders whatever tools the selector has approved. `ReActGuide` only appends Thought/Action/Observation framing to the system prompt, so its slot among the prompt-building guides is not load-bearing. Add custom guides with `builder.WithGuide<T>()`; they slot in before `HeadEvictionTrajectoryGuide`. After each guide (and the trajectory guide) runs, `DefaultGuideRunner` emits a `GuideContribution` via `ITracer.LogGuideContribution` — the structural delta that guide made to the draft (tools added/removed by name, memory snippets, system sections, trajectory messages, system-prompt char delta) — computed as a generic before/after diff so it works for every guide, including third-party ones, without the guide knowing about tracing. This surfaces the shaping decisions that never reach the final prompt (which tools a selector dropped, whether memory retrieval returned anything). `LogGuideContribution` is a default-interface no-op for back-compat; `ConsoleTracer`, `OpenTelemetryTracer`, and `CompositeTracer` implement it. A **supporting** guide that throws is caught by `DefaultGuideRunner`, skipped (its optional contribution lost for that turn), and logged via `ITracer.LogGuideError` (another default-interface no-op) — so a memory store timing out or a skill store failing to read degrades the turn's context rather than crashing the run. The mandatory trajectory guide is deliberately **not** wrapped: a turn with no rendered trajectory is worse than surfacing the failure. Model and tool calls are bracketed as **scopes**, not post-hoc events: the loop calls `ITracer.BeginModelCall`/`BeginToolCall`, gets an `IModelCallScope`/`IToolCallScope`, `Complete(response/result)`s it, then disposes it (`using`) — a scope disposed *without* `Complete` (the call threw) is recorded as a failed span. When a **model** call throws (after the SDK's own retries exhaust), the loop first calls `IModelCallScope.Fail(exception)` (a default-interface no-op) so the `chat` span carries the exception detail — `OpenTelemetryTracer` emits the standard OTel `exception` event (`exception.type`/`message`/`stacktrace`) + Error status instead of a bare "did not complete", `ConsoleTracer` emits a `model_call` line with the error; cancellation is excluded (expected control flow, not a call failure). Tool failures already carry detail — `ExecuteToolAsync` turns a throw into an `IsError` `ToolResult` that `Complete`s the span — so `Fail` is model-call-only. This lets `OpenTelemetryTracer` emit a nested OTel GenAI span tree (`invoke_agent` root → `chat {model}` / `execute_tool {name}` children carrying `gen_ai.*` attributes, plus the `gen_ai.client.token.usage` + `gen_ai.client.operation.duration` metrics) instead of flat events on one `agent.task` span; `ConsoleTracer` emits its `model_call`/`tool_call` JSON line on `Complete`; `CompositeTracer` fans one scope out to child scopes. Sensor and guide contributions stay event-shaped (`LogSensorResult`, `LogGuideContribution`, and `LogCompaction` — the last emitted by `DefaultGuideRunner` when the trajectory guide compacts, carrying a `CompactionTrace` of steps evicted / tokens reclaimed / folded / spend; `OpenTelemetryTracer` renders it as a `compact_context` event plus a `harness.compaction.tokens_reclaimed` histogram). Three more per-turn hooks close the loop's remaining blind spots — `LogRateLimit` (a rate-limit backoff wait), `LogCheckpoint` (each checkpoint save + its duration), and `LogBudgetSnapshot` (cumulative turns/tokens/cost/wall-clock vs the run's `Budget`, so burn-down is visible before the terminal `PartialResult`) — all default-interface no-ops emitted by `HarnessLoop`; `OpenTelemetryTracer` renders them as `harness.rate_limit`/`harness.checkpoint`/`harness.budget` events (plus `harness.ratelimit.wait` + `harness.checkpoint.duration` histograms), `ConsoleTracer` as `rate_limit`/`checkpoint`/`budget` JSON lines. `LogSensorResult` fires for **every** evaluation (pass, intervention, or error), not only interventions — `OpenTelemetryTracer` counts them all on a `harness.sensor.evaluations` counter (tagged by verdict) but only span-events the interventions/errors, so the root span doesn't bloat on passes. Every event and span carries a zero-based `turn` index so a backend can group all of a turn's work: the loop threads its turn counter into `BeginModelCall`/`BeginToolCall`/`LogSensorResult`, and `DefaultGuideRunner` derives the same index from the count of `ModelCallStep`s in the trajectory (the current turn's model call isn't appended until after guides run, so the count *is* the turn index). `ModelResponse` carries `Model`/`Provider` (populated by every adapter) to source `gen_ai.request.model`/`gen_ai.provider.name`, plus `CachedInputTokens`/`CacheWriteTokens` — the prompt-cache breakdown of `Usage.InputTokens` — which `OpenTelemetryTracer` splits into disjoint `cache_read`/`cache_creation` token types on `gen_ai.client.token.usage` (the `input` bucket becomes the uncached remainder) plus `gen_ai.usage.cache_read_input_tokens`/`cache_creation_input_tokens` span attributes, so cache hit-rate is observable; `ConsoleTracer` adds `cacheRead`/`cacheWrite` to its `model_call` line. `ModelResponse` also carries `InputTokensTowardRateLimit`, which is **not** the billed input: Anthropic excludes cache reads from ITPM, so `ClaudeModelClient` reports uncached input + cache writes and `TokensPerMinuteRateLimiter` counts that instead of the folded prompt — otherwise a well-cached agent throttles several times too early, and worse the better its cache performed. Adapters whose provider rules are not established leave it null and get the full prompt counted, which throttles early rather than late. The span-aware `ITracer` shape (`BeginModelCall`/`BeginToolCall` replacing `LogModelCall`/`LogToolCall`) is a breaking change for external tracer implementations.

`MemoryGuide` retrieves against the **latest** `UserMessageStep` (falling back to `TaskText` when the trajectory has none), not the frozen `TaskText`. The store does relevance ranking, so the query must describe what the agent is working on *now* — in multi-turn chat that's the current question, not the opener. This is unconditional, not a chat seam: a single-task run only ever has one `UserMessageStep` and its content *is* `TaskText` (nothing else appends one during an autonomous run), so task-mode behaviour is unchanged and no mode flag is needed.

Compaction is the one place a guide's output flows **back** to `AgentState` (guides are otherwise pure `ContextDraft` shapers). When `HeadEvictionTrajectoryGuide` evicts steps it calls the injected `ICompactionStrategy.CompactAsync(CompactionRequest, ct)` with the **typed** newly-evicted steps + the `PriorSummary`, and puts the `CompactionResult` on `ContextDraft.Compaction`; `DefaultContextBuilder` surfaces it on `ContextBuildResult`, and `HarnessLoop.CallModelAsync` commits `RollingSummary` (a fold persists it; a view returns null and keeps the prior) and accumulates `CompactionUsage`/`CompactionCost` onto the next state. A strategy returning `UpdatedSummary = null` is a stateless *view* (`NullCompactionStrategy` default, reconsiders the whole head each turn); a non-null summary is an incremental *fold* (`AiCompactionStrategy`, summarises only the new slice on the prior summary, so cost stays flat). The guide skips `RollingSummary.FoldedStepCount` head groups so already-folded steps are never re-summarised; the fold call is wrapped in try/catch (fail-open to an omission note) so a third-party strategy can't crash the run. The eviction **trigger** is `CompactionOptions.WindowTokens` — the per-turn context-window size the guide keeps each turn under — which is deliberately **separate** from `Budget.MaxTotalTokens` (the whole-run cumulative token cap the enforcer checks). Conflating them was a bug: eviction wants a single-window size, the enforcer wants a run total, and cumulative is always ≥ one window, so a shared field hard-stopped compacted runs at ~one window. The compaction opt-in methods (`WithAiCompaction(model, options)`, `WithCompactionStrategy<T>(options)`) **require** `CompactionOptions`, so a strategy is never wired without stating its trigger; a default (`WindowTokens = 100_000`) covers the base omission-note path, tunable via `WithCompactionOptions(...)`.

**Pinned reference content** (`ToolResult.Pins` → `AgentState.Pins` → `PinnedContextGuide`) protects foundational content from eviction. Head-eviction drops the *oldest* trajectory steps first, but the oldest are often the most load-bearing — a loaded procedure or output contract. An agent that loads its procedure via `skill_view` at turn 0 and enforces its output format would lose the contract to eviction, then fail to reproduce the format while an output sensor rejects it to the intervention cap (the sensor↔eviction trap: eviction drops the contract not knowing a sensor enforces it; the sensor rejects not knowing the contract was evicted). The fix: a tool returns `PinnedNote { Label, Content }` on `ToolResult.Pins`; the loop commits them to `AgentState.Pins` (replace-by-label) at execution time; `PinnedContextGuide` renders them as non-evictable **system sections** every turn — the same persistent region as the `[ORIGINAL GOAL]` pin and the tool catalogue, never touched by eviction/compaction. `SkillViewTool` uses it: it returns a short ack and pins the loaded skill body, so progressive disclosure (load on demand) is preserved *and* the loaded content survives compaction. This is the Letta/MemGPT core-memory-block pattern (labelled, in-context, persistent), written-on-load rather than agent-editable. `Pins` is checkpointed, so resume keeps loaded procedures. Pinned sections count toward the eviction window budget (they're real context), so a large pin correctly shrinks the room for episodic history.

### Sensor pattern
Sensors run **in parallel** at five hookpoints (`PreModelCall`, `PostModelCall`, `PreToolCall`, `PostToolCall`, `PreReturn`). A `SensorResult.Intervene(reason)` appends a `SensorInterventionStep` to the trajectory, which `HeadEvictionTrajectoryGuide` renders as a `[HARNESS OBSERVATION — ...]` assistant-role message on the next turn. `HarnessInstructionsGuide` primes the model in the system prompt to treat these notes as directives — feedforward and feedback working together. A sensor that **throws** fails open: `DefaultSensorRunner` catches it and returns `SensorResult.Failed(reason)` (flagged `IsError`, *not* an intervention), so one misbehaving sensor can't fault the parallel `Task.WhenAll` batch or take the run down, and the model keeps its turn. The failure is logged as an error via `LogSensorResult` (`OpenTelemetryTracer` tags the event `error.type=sensor_error` and does not count it as an intervention) rather than silently swallowed. A sensor that must fail **closed** (a hard security gate) should catch its own exception and return `Intervene` instead.

AI-powered sensors that call a model internally should return `SensorResult.PassWithUsage(usage, cost)` or `SensorResult.InterveneWithUsage(reason, usage, cost)` so the harness can accumulate the spend on `AgentState.SensorUsage` / `AgentState.SensorCost`. `DefaultBudgetEnforcer` includes these totals in its `MaxCost` and `MaxTotalTokens` checks.

Sensors may block actions but must never take turns away from the model — the model always gets the next call so it can self-correct. The loop decides what each intervention means per hookpoint:
- **PreModelCall**: **annotates** — injects the note into the trajectory then proceeds with the model call on the same turn; use for conditional pre-reasoning guidance (e.g. goal-drift warnings, error-streak alerts)
- **PostModelCall**: **rejects** — response is suppressed from trajectory so the model cannot re-see flagged content (e.g. PII); model gets a fresh turn to produce a clean response
- **PreReturn**: **challenges** — answer is not accepted; model gets a fresh turn with its prior answer visible so it can self-correct
- **PreToolCall**: **blocks** — tool is never dispatched; a `ToolCallStep` with `IsError = true` is recorded so the model sees a clean error and can replan
- **PostToolCall**: **flags** — advisory only; the tool has already run and its result is in the trajectory; intervention annotates it but cannot prevent the model from reasoning on it

### Structured output
`WithStructuredOutput<T>()` constrains the run's **final answer** to a type. It adds no port and no loop change — it is one guide plus one sensor over a shared `StructuredOutputContract<T>` (schema + `JsonSerializerOptions` + `TryBind`), so the thing that states the contract, the thing that enforces it, and the caller that reads it cannot drift.

- **`StructuredOutputGuide<T>`** contributes the schema to `ContextDraft.SystemSections`. That placement is load-bearing, not incidental: system sections are rebuilt from scratch every turn and `HeadEvictionTrajectoryGuide` only ever trims the *trajectory*, so the contract is unreachable by eviction **by construction**. This is the structured-output instance of the sensor↔eviction trap (see Pinned reference content); it needs no pin, because `Pins` exist to bridge *trajectory-borne* content (a `skill_view` result) into the persistent region, and a guide's contribution is already there. Pins are for content the model discovered; guides are for content you declared.
- **`StructuredOutputSensor<T>`** runs at `PreReturn` — the one hookpoint the loop reaches only on a turn with no tool calls, so it is *the final turn, known with certainty*. Every other framework broadcasts its output schema on every model call because a **generator** must decide before the call; a **validator** only needs to know after, and this loop already does. Intermediate reasoning turns are therefore left unconstrained. A final answer that does not bind is `InterveneWithToolSuppression`d: the binder's own error (which names the missing/malformed member) goes back as a `SensorInterventionStep`, the model gets a fresh turn to self-correct, and the loop's intervention cap bounds it to `PartialResult` rather than an exception. This repair loop is the differentiator — MEAI, Semantic Kernel, `Microsoft.Agents.AI`, the OpenAI Agents SDK, and the Vercel AI SDK all deserialize client-side post-hoc and either throw or hand back a `bool`, with no feedback to the model.
- **Binding is deliberately lenient where leniency is safe and strict where it is not.** `TryBind` strips markdown code fences and pulls the first balanced JSON value out of surrounding prose (weak models emit both regardless of instruction — Instructor treats this as a first-class transport, `Mode.MD_JSON`), and the default options inherit `JsonSerializerOptions.Web` for case-insensitive matching and numbers-from-strings. But it sets **`RespectRequiredConstructorParameters = true`** and `RespectNullableAnnotations = true`: without the former, System.Text.Json treats every constructor parameter as optional, `Deserialize<Triage>("{}")` yields `Triage { Category = null, Priority = 0 }` without throwing, and the validation silently becomes a rubber stamp. That flag is the whole reason the sensor is a validator and not a no-op; a unit test guards it.
- `[Description]` on the type and its members flows into the schema as `description` keywords, so the contract documents itself to the model. Non-object roots (`List<int>`, an enum) need no wrapping — the text channel carries any JSON value, unlike a tool-input schema, which must be an object.

The upgrade path if a model proves unreliable at emitting conforming text is **provider-native structured output in the adapters** (Ollama's `format` is grammar-constrained; Azure has `response_format: json_schema`; Anthropic has structured outputs), added as a default-interface method on `IModelClient` — the same back-compat idiom as `ITracer.LogGuideContribution`. It is *not* a "final answer" tool: that needs a terminal `ToolResult` flag, a loop change, an interaction with `ToolSelectorGuide` (which can drop the tool and strand the run), and a decision about co-emission with real tool calls. The sensor stays valuable even alongside native schema, because native is not a guarantee — MEAI's own docs say "Language models are not guaranteed to honor the requested schema," and every framework relying on it has no repair when it misses.

Sample: `samples/StructuredOutput`.

### State
`AgentState` is an immutable record. Every turn produces a new state via `with`-expressions. The trajectory (`IReadOnlyList<Step>`) is the append-only log of `UserMessageStep`, `ModelCallStep`, `ToolCallStep`, and `SensorInterventionStep`.

## Key ports

Defaults differ by entry point: `AddModelHarness` (bare, `Framework`) wires core + no-op defaults; `AddStandardModelHarness` (`Infrastructure`) calls `AddModelHarness` then overrides a few seams and adds the default tools/sensors. **Neither registers a model client — the caller always supplies one** via `.WithModel(...)` / `.WithResilientModel(...)`. Port defaults use `TryAdd` (a matching `.WithX(...)` replaces them); tools, sensors, and guides are additive.

`AddChatHarness` (`Framework`) is a third entry point for multi-turn chat — a thin sibling of `AddStandardModelHarness`, not a fork. It calls `AddModelHarness` then swaps two seams: `TurnScopedBudgetEnforcer` (counts turns/cost/tokens since the last `UserMessageStep`, so each user turn gets a fresh allowance instead of the whole conversation exhausting one budget) and `HeadEvictionTrajectoryGuide(pinOriginalGoal: false)` (drops the `[ORIGINAL GOAL]` pin, since chat's live goal is the latest user turn). No task-completion sensors are wired. Same `HarnessLoop`, `AgentState`, and `Agent`; continue a conversation with `AgentState.WithUserMessage`. Sample: `samples/Conversation`. (Multi-turn memory retrieval needs no chat seam — `MemoryGuide` already queries the latest user turn unconditionally; see the Guide pattern above.)

`AddStandardChatHarness` (`Infrastructure`) is the opinionated sibling — it calls `AddChatHarness` then adds chat-appropriate standard defaults: `InMemoryToolRegistry`, `GetDateTimeTool`, OpenTelemetry tracing, and the `PromptInjectionSensor` + `StuckDetector` security/loop sensors. It deliberately omits the task-completion `ProgressCheckSensor`. `PromptInjectionSensor` scans both inbound tool results (PostToolCall) and the latest user message (PreModelCall) — the latter checks the current chat turn, not just the opener. Sample: `samples/ChatSubAgent`.

The guide pipeline, `DefaultGuideRunner`, `DefaultSensorRunner`, `DefaultContextBuilder`, and `HeadEvictionTrajectoryGuide` are wired identically by both.

`TimeProvider` is registered as a singleton (`TimeProvider.System`) so every DI-resolved time-dependent component — `Agent`, `HarnessLoop`, the budget enforcers, the rate limiters, `ConsoleTracer`, `FileSkillStore` — reads one injectable clock; override it with standard DI (`services.Replace(...)`) rather than a dedicated builder method. Those components accept the clock by constructor injection (an optional `TimeProvider? = null` falling back to `TimeProvider.System`, except `Agent` which requires it), so direct construction still works without a container. `AgentState` is deliberately the exception: it stays a pure value type, so `NewTask` / `WithUserMessage` take a `DateTimeOffset timestamp` supplied by the caller (`Agent` passes `timeProvider.GetUtcNow()`) rather than depending on a clock at all.

## DI conventions

- DI registration files are named `DependencyInjection.cs`, class `DependencyInjection`, at the **project root** (not in a subfolder).
- Two-method pattern per abstraction: `AddXxx<T>()` (public, explicit override via `Replace`) and `AddXxxDefault()` (private, `TryAdd` — called only by `AddModelHarness`).
- Guides are a collection — `AddXxxGuideDefault()` uses `AddGuide<T>()` internally; opt-out by not calling the default.
- `AddModelHarness(systemPrompt)` is the single public aggregate entry point.

## Coding conventions

- SOLID principles and Clean / Onion Architecture with ports and adapters throughout.
- Primary constructors preferred.
- Interfaces over concrete implementations.
- Small, focused changes — build after each set of changes.
- Remove unused `using` statements in every file touched.
- No comments unless the WHY is non-obvious. XML doc comments belong on the public API surface only (this ships as a NuGet library, so consumers get IntelliSense) — keep them concise. Do not put XML docs on internal/private members; use a plain `//` comment there only where the WHY is non-obvious.
- No unnecessary abstractions — don't design for hypothetical future requirements.

## Workflow

- **Commit and push after every changeset** — do not wait to be asked.
- Before designing a non-trivial solution, ask clarifying questions to improve the spec.
- When using external SDKs or APIs, fetch up-to-date docs rather than relying on training data.

If a change renames a symbol, removes a project, alters a default, or adds/removes a feature, update every doc that referenced it so the docs never drift from the code.
